# Cloud Character API contract — Phase 1

This document locks the HTTP contract for a cloud character stored as a live snapshot. It deliberately excludes sharing, SSE, mutation queues, field-level merge, and conflict-resolution UI.

## General rules

- Authentication uses the existing HttpOnly session cookie.
- Request and response JSON use `camelCase`.
- Character IDs are UUIDs generated by the server.
- `localCharacterId` identifies the originating IndexedDB record for the authenticated owner.
- `deviceId` identifies the browser/device that created or updated the snapshot.
- The server calculates `contentHash`; clients never send or choose it.
- The server is the source of truth for `serverRevision`.
- Reads and mutations only expose active characters owned by the authenticated user in Phase 1.
- A character belonging to another user is reported as `404 CLOUD_CHARACTER_NOT_FOUND`.
- Delete is a soft delete and creates a tombstone through `deletedAt`.

## Snapshot fields

```json
{
  "name": "Lyra",
  "system": "daggerheart",
  "classKey": "wizard",
  "language": "pt-BR",
  "data": {
    "level": "1",
    "hp_current": "5"
  },
  "schemaVersion": 1
}
```

Allowed values:

- `system`: `daggerheart` or `custom`;
- `language`: `pt-BR` or `en-US`;
- `classKey`: one of the supported Daggerheart class keys when `system` is `daggerheart`; it must be `null` for `custom`;
- `schemaVersion`: positive integer, currently `1`.

## POST `/characters/cloud`

Publishes a local character as a cloud character.

Request:

```json
{
  "localCharacterId": "local-char-uuid",
  "deviceId": "device-uuid",
  "name": "Lyra",
  "system": "daggerheart",
  "classKey": "wizard",
  "language": "pt-BR",
  "data": {},
  "schemaVersion": 1
}
```

Success, new character: `201 Created`.

```json
{
  "character": {
    "id": "cloud-character-uuid",
    "ownerUserId": "owner-uuid",
    "localCharacterId": "local-char-uuid",
    "name": "Lyra",
    "system": "daggerheart",
    "classKey": "wizard",
    "language": "pt-BR",
    "data": {},
    "serverRevision": 1,
    "contentHash": "64-character-sha256",
    "schemaVersion": 1,
    "createdAt": "2026-07-09T12:00:00Z",
    "updatedAt": "2026-07-09T12:00:00Z",
    "deletedAt": null
  },
  "created": true,
  "reason": null
}
```

Idempotent retry with the same owner, `localCharacterId`, and identical normalized snapshot: `200 OK`, with `created: false` and `reason: "existing_identical_snapshot"`.

If the same active `localCharacterId` already points to a different snapshot, return `409 CLOUD_CHARACTER_ALREADY_EXISTS`. The response detail contains `characterId`, `localCharacterId`, and `serverRevision` so the frontend can reconcile explicitly.

## GET `/characters/cloud`

Lists active cloud characters owned by the current user. List items omit `data` and `deletedAt`.

Response: `200 OK`.

```json
{
  "characters": [
    {
      "id": "cloud-character-uuid",
      "ownerUserId": "owner-uuid",
      "localCharacterId": "local-char-uuid",
      "name": "Lyra",
      "system": "daggerheart",
      "classKey": "wizard",
      "language": "pt-BR",
      "serverRevision": 1,
      "contentHash": "64-character-sha256",
      "schemaVersion": 1,
      "createdAt": "2026-07-09T12:00:00Z",
      "updatedAt": "2026-07-09T12:00:00Z"
    }
  ]
}
```

## GET `/characters/cloud/{characterId}`

Returns the complete active snapshot for an owned character.

Response: `200 OK` with `{ "character": CloudCharacter }`.

Missing, deleted, or non-owned character: `404 CLOUD_CHARACTER_NOT_FOUND`.

## PATCH `/characters/cloud/{characterId}`

Replaces the mutable snapshot only when the request is based on the current server revision. Phase 1 intentionally sends the complete snapshot; Phase 4 can add the mutation form without removing this accepted shape.

Request:

```json
{
  "baseRevision": 1,
  "deviceId": "device-uuid",
  "name": "Lyra",
  "system": "daggerheart",
  "classKey": "wizard",
  "language": "pt-BR",
  "data": {
    "hp_current": "4"
  },
  "schemaVersion": 1
}
```

When `baseRevision` equals the current `serverRevision` and content changed, the server stores the snapshot and increments the revision.

Response: `200 OK`.

```json
{
  "character": {},
  "unchanged": false
}
```

If the normalized snapshot has the same `contentHash`, the server does not increment the revision and returns `unchanged: true`.

When `baseRevision` is stale, no data is changed and the server returns `409 REVISION_MISMATCH`:

```json
{
  "code": "REVISION_MISMATCH",
  "message": "The cloud character changed after this snapshot was loaded.",
  "detail": {
    "characterId": "cloud-character-uuid",
    "serverRevision": 3,
    "receivedBaseRevision": 1
  }
}
```

Phase 1 never performs last-write-wins and never merges stale snapshots.

## DELETE `/characters/cloud/{characterId}`

Soft-deletes an owned character.

Response: `200 OK`.

```json
{
  "ok": true,
  "characterId": "cloud-character-uuid",
  "deletedAt": "2026-07-09T12:00:00Z"
}
```

A repeated delete or a non-owned ID returns `404 CLOUD_CHARACTER_NOT_FOUND` in Phase 1.

## Error codes

| HTTP | Code | Meaning |
| --- | --- | --- |
| 401 | `SESSION_EXPIRED` | Existing session cookie is absent or invalid. |
| 404 | `CLOUD_CHARACTER_NOT_FOUND` | Character is missing, deleted, or not owned by the caller. |
| 409 | `CLOUD_CHARACTER_ALREADY_EXISTS` | The active `localCharacterId` is already attached to a different cloud snapshot. |
| 409 | `REVISION_MISMATCH` | `baseRevision` is older or newer than the current revision. |
| 413 | `CHARACTER_TOO_LARGE` | Encoded normalized snapshot exceeds the configured limit. |
| 422 | `UNSUPPORTED_CHARACTER_SCHEMA_VERSION` | `schemaVersion` is not supported by the server. |
| 422 | `INVALID_CHARACTER_SNAPSHOT` | Snapshot violates a domain rule after normalization. |

FastAPI's standard validation response may also be returned for malformed JSON or field-level type/length errors.
